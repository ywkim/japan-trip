# 대시보드 대안 검토

`viz/dashboard.html` (327줄, 인라인 DECISION 데이터 + 슬라이더 + 점수 입력 + localStorage)
이 정말 필요한지, 더 가벼운 대안은 무엇인지 정리한다.

## 현재 대시보드가 하는 일

| 기능 | 설명 | 대체 가능성 |
|---|---|---|
| 가중치 슬라이더 | 7개 기준 가중치를 실시간 조정 → 순위 즉시 반영 | **어려움** — 인터랙티브 민감도가 핵심 가치 |
| 점수 입력 폼 | 후보 × 기준 1~10 점수 입력 (localStorage 저장) | 쉬움 — `data/decision.json` 직접 수정으로 대체 |
| 순위 시각화 | 막대그래프 + 종합 점수 + coverage % | 쉬움 — `score.py` 출력 / markdown 표 |
| JSON 내보내기 | 현재 상태를 파일로 다운로드 | 불필요 — 정본은 `decision.json` |

## 비용 (현재 구조의 실제 부담)

1. **이중 데이터 소스**
   - `data/decision.json` (정본) ↔ `viz/dashboard.html` 내 인라인 `DECISION` 객체
   - CLAUDE.md에 "JSON 수정 시 HTML도 동시 갱신 필요" 규칙이 명시됨 → 사람·AI 모두 잊기 쉬움
2. **localStorage 격리**
   - 영욱·소연 각자의 브라우저에 별도 저장 → 상대 입력이 보이지 않음
   - 즉, **협업 도구가 아닌 개인 메모장**으로만 작동
   - 결과물이 git에 들어가지 않으면 누적 자산이 되지 않음 (CLAUDE.md 메타 규칙과 충돌)
3. **두 개의 입력 경로**
   - JSON 편집 vs 대시보드 입력 → 어느 쪽이 진실인지 모호
   - 현재 교토 점수만 JSON에 들어가 있고, 다른 후보는 대시보드에 입력해도 PR이 되지 않음
4. **유지보수 표면적**
   - 327줄 HTML/JS — 후보·기준·필드 추가 시 두 곳 수정

## 대안 옵션

### A. 현상 유지

- **장점**: 즉시 작동, 작업 비용 없음
- **단점**: 위 4가지 부담 그대로

### B. 읽기 전용 민감도 대시보드 (권장)

- 입력 UI·localStorage·내보내기 **제거**
- 가중치 슬라이더 + 순위 시각화만 유지
- 인라인 데이터는 `scripts/build-dashboard.py`가 `decision.json`에서 빌드 시 주입
  → 수동 동기화 규칙 제거
- **장점**: 핵심 가치(슬라이더 민감도) 유지 + 단일 진실 출처 회복
- **단점**: 빌드 스크립트 1개 추가, 점수 변경은 PR로만 (의도된 제약)

### C. CLI 민감도 보고서

- `python scripts/score.py --sensitivity` → 각 기준의 가중치를 ±0.05 흔들었을 때 순위 변화표 출력
- markdown 또는 텍스트로 PR에 첨부 가능
- **장점**: PR 리뷰 가능, HTML 없음, 즉시 재현 가능
- **단점**: 슬라이더만큼 직관적이지 않음 — "한 손으로 굴려보기"가 안 됨

### D. 대시보드 폐기 (markdown + CLI only)

- 결과는 `python scripts/score.py`, 비교는 markdown 표로
- **장점**: 가장 단순, 유지비 0
- **단점**: 가중치 의견이 갈렸을 때 토론 도구 없음 — MCDA의 핵심 약점

### E. Jupyter 노트북

- 슬라이더 위젯(ipywidgets) + 차트
- **단점**: 더블클릭으로 안 열림, 런타임 필요, 부부 둘 모두에게 과한 도구
- **결론**: 이 레포의 사용자(부부 2인)에게 부적합

### F. 외부 SaaS (Google Sheets / Notion DB)

- 협업·실시간 동기화는 자연스러움
- **단점**: 레포 밖으로 데이터가 나감 → "단일 출처는 `decision.json`" 원칙 깨짐 + 누적 자산 분산

## 권장: B + C 하이브리드

| 산출물 | 역할 |
|---|---|
| `data/decision.json` | **단일 입력**. 점수·가중치 변경은 모두 여기서 PR로 |
| `viz/dashboard.html` (B로 축소) | **인터랙티브 민감도 토론용**. 빌드 산출물, 수동 편집 금지 |
| `scripts/score.py --sensitivity` (C 추가) | **PR 첨부용 정적 보고서**. "weight ±0.05 시 1·2위 안 바뀜" 같은 결론을 텍스트로 |
| `scripts/build-dashboard.py` | `decision.json` → 대시보드 인라인 데이터 주입. CI 또는 pre-commit에서 자동 |

### 이 구성이 해결하는 것

- **이중 데이터 소스 → 빌드 자동화로 해소**
- **localStorage 협업 부재 → 입력 UI 제거로 문제 자체 소거** (협업은 PR로)
- **두 개의 입력 경로 → 하나로 통일**
- **MCDA 가중치 토론 가치 → 슬라이더 유지로 보존**

### 이 구성이 포기하는 것

- 브라우저에서 점수를 바로 입력하는 편의 — 의도된 트레이드오프
- 즉, "지금 점수를 즉흥적으로 바꿔보고 싶다"면 슬라이더로 가중치만 흔들면 충분하고,
  점수 자체를 바꾸려면 **commit 의도가 있는 변경**으로 격상된다

## 결정 메모 (제안)

- **합의 후보**: B + C 하이브리드로 다음 PR에서 축소·정리
- **보류**: 단순 D(폐기)로 가도 되는지 — 가중치 슬라이더가 실제로 부부 토론에 쓰였는지 1회 사용해본 뒤 결정
- **즉시 가능한 최소 변경**: `scripts/score.py --sensitivity`만 먼저 추가 (대시보드는 손대지 않음) → 효용 검증 후 B 진행
