# 2026-05-31 — 카페·식당 Maps 링크: 가게명+지점 검색, 모호 케이스는 Place ID(CID)

## Status

Accepted

## Context (왜)

PR #89(6/1 eX cafe·만자라테이 번역 페이지)에 각 가게의 위치 확인용 Google Maps 링크를 추가하는 과정에서, 링크 방식에 따라 결과가 달라지는 문제가 드러났다(사용자 스크린샷·지적, 2026-05-31).

- **주소만으로 검색**(`maps/search/<주소>`): 좌표는 정확히 찍히나, 폰토초·아라시야마역 같은 **밀집 구역에서는 구글이 건물 중심 좌표에 가장 가까운 인접 POI의 카드를 표시**한다. 만자라테이(`下樵木町198-1`) 링크가 옆 가게(블루보틀 커피)를 카드로 띄워 "같은 주소에 다른 가게?"라는 혼동을 유발. eX cafe는 독립 건물 POI라 주소=가게 좌표가 일치해 정상.
- **가게명만으로 검색**: 카드는 뜨지만 **복수 지점 브랜드(% Arabica·雲ノ茶 등)는 다른 지점에 안착**할 위험. % Arabica는 嵐山(글로벌 플래그십)·東山 등 교토 내 복수 지점 존재(아라시야마 ftid `0x…:0xec55dccfaa9eb7c0` vs 東山 ftid `0x…:0x5241bd15bb462f3b` — 완전히 다른 장소).

사용자 질문 1: "주소 좌표·가게 둘 중 하나만 가능한가?" → 아니다. 둘 다 가능.
사용자 질문 2: "완전 무결한 방식(Place ID)을 실제로 시도했나?" → 최초엔 말로만 언급하고 미시도. 본 결정에서 실제 시도.

## Decision (무엇)

가게의 **고유 식별 강도에 맞춰 Maps 링크 방식을 차등 적용**한다. 보이는 주소 텍스트는 모두 검증된 일본어 주소를 그대로 유지(읽기용 정확 주소), **링크 href만** 아래로 설정.

1. **고유 상호(지점 포함) → 가게명+지점 검색**: 일본 가게는 상호에 지점명(`嵐山本店`·`嵐山店`·`先斗町店`·`嵐山庭園`)을 포함하므로, 가게명+지점명으로 검색하면 정확한 그 POI에 매칭 → 카드와 좌표를 동시에 얻는다.
   - eX cafe → `eX cafe 京都嵐山本店`
   - 嵯峨野湯 → `嵯峨野湯 嵐山`
   - パンとエスプレッソと → `パンとエスプレッソと嵐山庭園`
   - 雲ノ茶 → `雲ノ茶 嵐山店`
   - まんざら亭 先斗町店 → `まんざら亭 先斗町店`
2. **질의로도 모호한 케이스(% Arabica) → Place ID(CID) 직링크**: 완전 무결한 방식을 실제 적용. `https://maps.google.com/?cid=17029760350450268096` (아라시야마 글로벌 플래그십, 좌표 35.0135422,135.6764367 — 도게츠교 상류 강변, 검증됨). CID는 가게 고유 ID라 이름 중복에 완전 면역.

### Place ID 시도 기록 (실제 시도한 절차)

- `WebFetch`로 Google Maps 페이지 직접 파싱 → **실패**(JS 렌더링·스크래핑 차단, ID 추출 불가).
- `WebSearch`로 우회 → 구글 **공식 canonical place URL**이 결과에 노출되어 거기서 feature ID(ftid) `0x6001a9ffffef6009:0xec55dccfaa9eb7c0` 확보.
- ftid의 두 번째 hex(`0xec55dccfaa9eb7c0`)를 10진 CID(`17029760350450268096`)로 변환 → `?cid=` 직링크 구성.
- 한계: Place ID/CID는 가게마다 개별 조회가 필요하고 headless fetch로는 일괄 추출 불가(WebSearch가 canonical URL을 노출해야만 획득). 따라서 **이름이 이미 고유한 가게엔 과한 작업** — 모호 케이스(% Arabica)에만 적용.

채택하지 않은 대안:
- **모든 가게 CID 직링크**: 일괄 ID 조회가 비싸고(가게마다 WebSearch), 이름이 고유한 가게엔 이득 없음. 미채택.
- **모든 가게 주소 검색 유지**: 밀집 구역 인접 POI 카드 오표시 문제 미해결. 미채택.
- **모든 가게 가게명 검색(지점 미포함)**: 복수 지점 브랜드 오안착 위험. 미채택.

## Consequences (그래서)

**긍정**:
- 모든 카페·식당 링크가 **정확한 가게 카드 + 정확한 좌표** 둘 다 안착. 가족이 탭 시 헷갈리지 않음.
- % Arabica는 CID로 지점 오안착 위험 원천 차단(완전 무결 방식 실제 적용).
- 보이는 주소 텍스트는 검증된 일본어 주소 유지 — 읽기용 정확성도 보존.

**부정·트레이드오프**:
- 링크 방식이 가게별로 2종(가게명+지점 / CID)으로 갈림 — 단, 가게 식별 강도에 따른 의도된 차등.
- CID는 가게가 구글에서 통폐합·재등록되면 깨질 수 있음(드묾). 깨지면 가게명+지점 검색으로 폴백.

**후속 행동**:
- 향후 신규 가게 Maps 링크: ① 상호가 지점까지 고유하면 가게명+지점 검색, ② 질의로도 모호(복수 지점)하면 WebSearch로 canonical ftid 확보 → CID 직링크.

**영향 받은 파일**:
